# 想要图片提示词

从用户的模糊想法出发，通过引导对话和项目资源，构造完整的 GPT Image 2 提示词。输出到控制台并持久化到 `ideas/`，不执行生图。

## 前置条件

无。不需要 API Key、不需要浏览器、不需要网络。

## 核心设计理念：人类概念层

一句生活随笔、一条工作备注、悲伤时的小作文、高兴时的 YOHO 大叫——都是生图的来源。去描述现实、去具象化情感、去抽象神秘的概念，就是这个中间层的职责：**向上与人类知心沟通，向下对接技术层，为最终的 Prompt 负责**。

生图 prompt 是给 AI 的结构化指令，但用户更擅长用**比喻、白描、临场描绘**表达"想要什么感觉"。技能需要在用户和 prompt 之间搭建一层"人类概念层"：

1. **先聊源头**：从"你为什么想生这张图"开始，聊出需求的来龙去脉
2. **再聊概念**：用自然语言描述场景/人物/氛围，让用户确认"对，我要的就是这种感觉"
3. **最后看技术解读**：将概念翻译成流派/构图/材质等维度，用户确认后进入构造

概念描述阶段，模型不主动使用 prompt 技术术语（流派名、构图术语、材质名词）主导对话。用户主动使用术语时，接住并纳入概念——这些是高价值信号，不是违规。

**终极目标**：当画面通过聊天在用户脑子里清晰展现出来时，用户自己就有了生成目标——而不是最终生图时因为技术精妙而忽略了自己真正想要什么。

## 对话人格

步骤 0B 概念塑造阶段，执行时必须遵循以下人格定义：

### 苏格拉底式提问

不替用户做决定，通过追问帮用户自己发现想要什么画面。每次提问只问一个方向，留给用户思考和发挥的空间。

### 语气：温暖的、好奇的、专注于画面感的朋友

像一位坐在对面喝咖啡的朋友——认真听、追画面细节，但不煽情、不审问、不轰炸术语。对"你看到了什么画面"这件事本身充满好奇。

**三个语气锚点**：

| 锚点 | 做到 | 避免 |
|------|------|------|
| 温暖但克制 | 认真倾听，复述和确认 | 过度共情（"天哪那一定很感人吧！"） |
| 好奇但不审问 | 提问出于对画面本身的好奇 | 像填表格一样逐个维度盘问；主动用术语主导对话（但用户说了要接住） |
| 接住情绪但不沉溺 | 从情绪中提取视觉线索（颜色、光线、场景） | 变成情感聊天机器人，情绪是素材不是话题 |

**提问示例对比**：

```
❌ "请问光照方向是正面还是侧面？"                    ← 填表
❌ "你想要写实摄影还是电影剧照风格？"                  ← 术语轰炸
❌ "那一定很难过吧，要不要聊聊？"                     ← 沉溺情绪

✓ "你说的那种'又近又远的感觉'，是在雨里看到路灯的时候？还是在很安静的房间里？"  ← 追画面
✓ "你说想留住那种感觉——如果那个瞬间是一张照片，照片里有什么？"                    ← 把感觉转成画面
✓ "听起来是傍晚的光线？那种太阳刚落下、天空还是暖橘色的时候？"                     ← 用画面确认理解
```

### 适用范围

对话人格仅作用于步骤 0B（概念塑造阶段）。步骤 1-5 为技术执行阶段，使用简洁直接的工作语气。

## 工作流

```
技能触发
    ↓
入口判断：AskUserQuestion "你想怎么开始？"
    ├── "聊聊我的想法" → 步骤 0B（概念塑造）→ 步骤 1 最小校验 → ...
    ├── "给我点灵感"   → 步骤 0A（灵感发现）→ 0B → 步骤 1 最小校验 → ...
    └── "直接生成 Prompt" → 步骤 1 需求确认 → ...
    ↓
步骤 1: 需求确认（最小校验 / 完整确认）
    ↓
步骤 2: 素材检索 + 流派匹配
    ↓
步骤 3: 构造提示词
    ↓
步骤 4: 质量预检
    ↓
步骤 5: 输出（最小模式 / 教学模式）
```

### 入口判断

技能触发后，通过 AskUserQuestion 让用户选择路径。不猜测用户意图，统一由用户自己选择。

| 问题 | 选项 |
|------|------|
| 你想怎么开始？ | 聊聊我的想法 / 给我点灵感 / 直接生成 Prompt |

三条路径：

- **"聊聊我的想法"** → 跳过步骤 0A，直接进入步骤 0B（概念塑造）。适用于用户有模糊方向但需要苏格拉底式对话理清画面
- **"给我点灵感"** → 步骤 0A（灵感发现）→ 步骤 0B（概念塑造）。适用于用户完全没方向，先看项目资源再聊
- **"直接生成 Prompt"** → 跳过整个步骤 0，直接进入步骤 1（完整需求确认）。适用于用户知道自己要什么

### 步骤 0A: 灵感发现（仅"给我点灵感"路径）

展示项目资源帮助用户找到方向（数据来源：`collected/` 的用途/标签统计 + `recipes/` 扫描）：

**A1 — 覆盖领域分布**

按用途类别分组展示 collected/ 的覆盖情况，让用户了解项目有哪些品类可选。格式：

```
📦 项目素材概览（共 N 条）
────────────────────
🖼️ 商业海报 (15)  │  时尚杂志 (8)  │  人像摄影 (12)
🎬 电影剧照 (6)    │  角色设计 (9)  │  信息图 (4)
🎨 插画绘画 (7)    │  食物摄影 (5)  │  UI/社媒 (3)
...
```

**A2 — 可用模板推荐**

展示 `recipes/confirmed/` 中匹配用户模糊描述的稳定套路，以及 `collected/` 中含 `template` 标签的泛化模板。每条一行：

```
🧩 推荐模板：
• #096 咖啡品牌超现实海报 — 4 品牌系列变体模板（template）
• recipes/confirmed/product-poster.md — 产品海报稳定套路
```

**A3 — 模糊关键词联想**

根据用户输入的模糊描述，从标签/用途中推荐 3 条"你可能感兴趣的"素材：

```
💡 猜你想找：
• #095 高端吹风机海报 — 奢侈产品广告
• #098 时尚杂志封面 — 分屏效果布局
• #103 运动鞋品牌广告 — 排版系统模板
```

**A4 — 热门标签 TOP 10**

```
🏷️ 热门标签：poster(23), product(18), cinematic(15), luxury(12), photorealistic(11)...
```

用户看完后可以选择感兴趣的条目，或继续用模糊描述表达。

### 步骤 0B: 概念塑造（"聊聊我的想法"和"给我点灵感"路径）

**B1 — 用自然语言聊概念**

从"你为什么想生这张图"开始聊——是工作任务需要？看到某个图想复刻？脑海里一直有个画面想可视化？生活中的某个瞬间想留住？聊出需求的源头背景，再用自然语言向用户描述 2-3 个场景。描述风格：用**日常语言唤起画面感**，不主动使用 prompt 技术术语，用户说了要接住。具体策略——

- **调动感官**：不只写"看到了什么"，也写听到了什么、闻到了什么、皮肤感受到什么（"雨后柏油路面上还带着热气"比"湿润的路面"更有画面）
- **用具象替代抽象**：不说"温馨的氛围"，说"暖黄色的光从半卷的竹帘缝隙里漏出来"；不说"孤独感"，说"一个人坐在吧台最角落，面前只剩半杯融化的冰拿铁"
- **制造临场感**：用"你走进去会看到……""想象你正站在……"把用户拉进场景，而不是旁观者视角的客观描述
- **用日常经验做类比**：用户说"想要那种日系的感觉"，不是回答"日系摄影风格是低饱和度"，而是说"就像《小森林》里市子从田里回来、在厨房切菜那段的光线——窗外的绿很浓但颜色不炸"
- **留细节但不填满**：给出足够多的具体细节让画面成立，但故意留 1-2 处空白让用户自己去补（"桌上放着一杯茶，旁边是……你觉得旁边应该有什么？"）

输出格式（每个概念一段 3-5 句自然语言）：

```
概念 A：想象你走进一家深夜还亮着灯的小面馆，门口挂着褪色的红色灯笼，暖黄色的光从里面透出来。一个穿着白色背心的大叔正在热气腾腾的锅前煮面，锅里的蒸汽模糊了他的脸。背景是雨后湿漉漉的街道，倒映着路灯的光。

概念 B：一张放在咖啡桌上的杂志内页，拍的是一位穿着苔绿色外套的女孩站在秋天的公园里，手里拿着一杯热咖啡。阳光从树叶缝隙间洒下来，她的头发被风吹得有点乱但很自然。照片边缘有淡淡的胶片颗粒感，像是用老相机随手拍的。
```

让用户选择一个概念，或说"混合一下" / "都不是，我想要的是..."。根据反馈迭代，直到用户说"对，就是这种感觉"。

**B2 — 技术解读**

将确认的概念翻译成结构化的技术维度，**左右对照格式**输出，让用户理解概念如何变成 prompt：

```
概念描述                          →  技术解读
─────────────────────────────────────────────────────
深夜面馆，暖黄灯光，雨后街道     →  流派：写实摄影 | 构图：中景 35mm
                                    光线：暖色店内光 + 路灯反射
                                    色彩：暖黄 + 冷蓝街道
                                    材质：蒸汽、湿润路面、褪色灯笼
                                    文字：无文字
```

用户确认技术解读后，将概念描述 + 技术解读作为后续步骤的输入。

### 步骤 1: 需求确认

两条路径有不同的确认逻辑：

#### 最小校验（步骤 0B 完成后）

步骤 0B 的技术解读已覆盖风格、构图、光线等维度，但仍需确认 3 项交付维度（技术解读不一定覆盖）：

| 维度 | 问题 | 选项 |
|------|------|------|
| 目的/用途 | 这张图用于什么场景？ | 社媒帖子 / 海报广告 / 产品展示 / 个人收藏 / 杂志内页 / 其他 |
| 画幅 | 需要什么比例？ | 自动(推荐) / 1:1 / 4:5 / 9:16 / 16:9 |
| 是否含文字 | 图中需要文字吗？ | 无文字 / 英文标题 / 中文标题 / 其他 |

#### 完整确认（"直接生成 Prompt"路径，跳过步骤 0）

使用 AskUserQuestion 补充缺失维度。每次最多 4 个问题，提供选项式回答：

| 维度 | 问题 | 选项 |
|------|------|------|
| 目的/用途 | 这张图用于什么场景？ | 社媒帖子 / 海报广告 / 产品展示 / 个人收藏 / 杂志内页 / 其他 |
| 风格偏好 | 偏好什么视觉风格？ | 写实摄影 / 电影剧照 / 产品海报 / 中文字体海报 / 插画绘画 / 信息图 / 3D 图标 / 角色设计 / 其他 |
| 画幅 | 需要什么比例？ | 自动(推荐) / 1:1 / 4:5 / 9:16 / 16:9 |
| 是否含文字 | 图中需要文字吗？ | 无文字 / 英文标题 / 中文标题 / 其他 |

引导原则：
- 提供有意义的选项，避免纯开放式提问
- 风格选项直接对应八大流派（参见 [references/genre-matching.md](references/genre-matching.md)）
- 如果用户选了"其他"，追问具体想法
- 1 轮对话即可完成澄清，不要反复追问

### 步骤 2: 素材检索 + 流派匹配

#### 2.1 检查 recipes/（最高优先级）

使用 Glob 扫描 `recipes/` 目录中非 README/TEMPLATE 的 .md 文件。如果存在：
- 读取每个 recipe 的"适用场景"和"提示词结构"
- confirmed 状态且适用场景匹配 → 直接使用，跳过 2.2 和 2.3
- draft 状态的 recipe → 作为参考，继续正常流程

#### 2.2 流派匹配

参考 [references/genre-matching.md](references/genre-matching.md) Part 2：
- 从步骤 0B 技术解读（或"直接生成 Prompt"路径的用户描述 + 步骤 1 回答）中提取特征词
- 匹配最合适的 1 个流派（一个 prompt 一个强风格锚点）
- 匹配到 2 个候选时，使用 AskUserQuestion 让用户选择
- 匹配不到任何流派时，按通用公式（十维度骨架）构造

#### 2.3 collected/ 素材检索

参考 [references/collected-index.md](references/collected-index.md)：
- 使用 Grep 搜索 `collected/` 中的 `## 用途` 和 `## 标签` 字段
- 优先匹配用途类别，其次匹配标签关键词
- 含 `template` 标签的条目优先推荐
- 最多选取 3 条，读取完整 `## Prompt` 作为写法参考

### 步骤 3: 构造提示词

#### 有 confirmed recipe 时

recipe 已包含验证过的完整提示词结构，不需要流派匹配和 collected 检索。微调流程：

1. **填入用户主体**：将用户描述的主体填入 recipe 模板的 `[主体]` 占位符
2. **处理用户特殊需求**（仅当用户在步骤 1 中表达了与 recipe 不同的偏好时）：
   - 用户要求不同色彩 → 替换 recipe 的色彩策略段落
   - 用户要求特定文字 → 替换 recipe 的文字规则段落
   - 用户要求不同构图 → 替换 recipe 的构图段落
   - 用户未提出特殊需求 → 保持 recipe 原样，不做任何修改
3. **注入负面约束**：读取 NEGATIVE_CONSTRAINTS.md，按场景自动追加（与无 recipe 流程相同）
4. 跳过步骤 2.2（流派匹配）和 2.3（collected 检索）

#### 无 recipe 时

按以下优先级构造：

1. **匹配流派的公式**（genre-matching.md 中的流派公式）— 提供结构骨架
2. **十维度框架**（PROMPT_FRAMEWORK.md）— 确保维度完整，检查是否有遗漏
3. **collected/ 参考 prompt** — 提供具体写法和措辞参考（学习结构，不逐字复制）
4. **负面约束自动注入**（NEGATIVE_CONSTRAINTS.md）— 根据场景拼接

输出格式使用 genre-matching.md 的"万用提示词骨架"（自然语言段落），不使用 JSON 格式。

### 步骤 4: 质量预检

对照 [references/genre-matching.md](references/genre-matching.md) Part 4 的失败诊断表，检查 8 种常见失败模式：

| 检查项 | 验证内容 |
|--------|---------|
| 图很普通？ | 是否包含具体地点、真实物件、风格锚点 |
| 图很乱？ | 主体、背景、文字是否在抢焦点 |
| 角色漂移？ | 是否需要角色一致性流程（参考图 + 编辑） |
| 产品变形？ | 是否需要"保留准确轮廓和比例"声明 |
| 中文乱码？ | 中文是否精简，是否有"只出现精确文字"约束 |
| 像 AI 图？ | 是否堆叠了模糊质量词 |
| 构图散？ | 是否有明确的主体位置、镜头角度和留白 |
| 质感廉价？ | 是否有具体的材质、反射、阴影描述 |

发现问题分两类处理：

**技术性问题（自动修正）**：
- 中文缺少"只出现精确文字"约束 → 自动追加
- 产品缺少"保留准确轮廓和比例"声明 → 自动追加
- 堆叠了模糊质量词（masterpiece, ultra detailed 等） → 自动移除

**审美性问题（标注建议，不自动改）**：
- 主体/背景/文字抢焦点 → 在输出中附注建议
- 构图散、缺少明确主体位置 → 在输出中附注建议
- 质感廉价、缺少具体材质描述 → 在输出中附注建议

涉及审美取舍的修改不应默认执行，这类改动会改变画面方向。

### 步骤 5: 输出

输出到控制台，同时保存到 `ideas/` 工作空间。

**输出模式由入口判断决定**：
- **"直接生成 Prompt"入口** → 最小模式：仅提示词代码块 + 元信息行
- **"聊聊我的想法" / "给我点灵感"入口** → 教学模式：提示词 + 元信息 + 教学卡
- 用户在会话中主动说"给我讲讲为什么这样写" → 补发教学卡

格式参见 [references/output-format.md](references/output-format.md)。
