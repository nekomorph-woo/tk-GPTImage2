# 研究：参考图驱动的角色一致性

- **研究日期**：2026-05-04
- **研究方向**：使用参考图（--reference）实现 GPT Image 2 角色一致性
- **选择理由**：用户需要碧蓝航线角色 95%+ 一致性生图，参考图是最直接的技术路径。recipes/ 当前为空，该方向无 confirmed recipe。collected/ 有 #076 角色设定集相关素材。
- **涉及信息源**：OpenAI 官方文档、OpenAI 社区、项目魔导书、社区画廊（共 45 条素材）

## 来源

### 来源：OpenAI 官方文档

根据 [01] 的指导：

**核心原则**
- 身份保留是 gpt-image-2 的六大核心能力之一，参考图（Reference Image）是实现角色一致性的基础机制（来源 [01]）
- 角色锚点（Character Anchor）是官方推荐的标准两步工作流：先创建参考图锁定外貌/比例/服装/基调，再用编辑模式在新场景中重用角色（来源 [01]）
- 每轮迭代必须重申不变量（Restate Invariants），编辑时使用 "change only X" + "keep everything else the same" 模式（来源 [01]）
- 迭代优于过载（Iterate Instead of Overloading）：长 prompt 可行，但调试时应从干净基准 prompt 开始，逐步细化（来源 [01]）

**技巧模式**
- 虚拟试穿的身份锁定模式：明确锁定面部、体型、姿势、发型、表情，仅允许改变服装（来源 [01]）
- 多图输入按索引引用："Image 1: product photo... Image 2: style reference..."（来源 [01]）
- 显式约束声明：列出排除项和不变量（"no watermark"、"preserve identity/geometry/layout/brand elements"）（来源 [01]）
- 身份敏感场景建议使用 quality="medium" 或 quality="high"（来源 [01]）

**反面案例**
- 在长对话中累积生成而不重申不变量，会导致角色特征漂移（来源 [01]）
- 一次改变过多变量，难以定位哪个改动导致了偏移（来源 [01]）

**可操作项**
- 创建角色参考图时使用纯净背景，避免场景元素干扰模型对角色的提取（来源 [01]）
- 编辑时显式列出"不变量清单"：哪些特征必须保留、哪些允许改变（来源 [01]）

### 来源：OpenAI 社区（DNA Template 与稳定模板）

根据 [02][03] 的指导：

**核心原则**
- DNA Template 是社区验证最有效的角色一致性文本技巧：在每次生成新场景前发送完整的角色描述模板，锁定所有视觉特征、风格和基调（来源 [02]）
- 结构化描述模板比随机拼凑的 prompt 稳定得多，固定顺序（类型声明->头发->眼睛->性格->体型->服装->姿势->构图约束->光线->质量指令）能有效避免解剖比例不一致和表情丢失（来源 [03]）
- 显式一致性声明——"Their appearance must remain perfectly consistent in every image"——是不可省略的（来源 [02]）

**技巧模式**
- DNA Template 的核心要素包含：画面格式/比例、艺术风格、项目标题/主题、完整角色外观（从头到脚）、情绪基调、风格渲染指令、构图规范、显式一致性声明（来源 [02]）
- 多角色差异化标记：为相似角色分配独特的配饰作为身份区分标识（来源 [02]）
- 防止多面板/参考表输出：在 prompt 中声明 "single character, no reference sheet, no multiple panels"（来源 [03]）
- DNA Template 在每次生成前重发，让模型"重新校准"角色理解（来源 [02]）

**反面案例**
- 社区报告的主要痛点：肤色/体型变化、艺术风格和色调漂移、AI 解释覆盖用户意图（来源 [02]）
- 在 prompt 中出现 "reference sheet" 或 "multiple views" 词汇会触发模型自行展开为角色设计表（来源 [03]）

**可操作项**
- 将角色外貌描述按固定结构组织，形成可复用的 DNA Template（来源 [02][03]）
- 对碧蓝航线角色，可以编写"角色 DNA"锁定发型、发色、眼睛、服装、配饰，在每次生成前重发（来源 [02]）

### 来源：项目魔导书

根据 [04] 的指导：

**核心原则**
- 两步工作流是角色一致性的核心范式：先做参考图（generate），再做场景编辑（edit）（来源 [04]）
- "一个提示词只放一个强风格锚点"：避免混合多种风格导致输出不稳定（来源 [04]）
- 迭代原则：一次只改一个变量（光线、构图、色彩、材质、文字），不要一起大改（来源 [04]）

**技巧模式**
- 参考图 prompt 的关键要求：明确角色名字和身份、纯净背景（暖灰色摄影棚）、标准站姿（三分之四）、完整服装和配饰描述、柔和棚拍光、"不要额外配饰，不要文字"（来源 [04]）
- 场景编辑 prompt 的关键要求：开头声明"使用参考角色"、列出需要保留的具体特征、明确"保持身份和服装一致"、限定"只改变环境和姿势"（来源 [04]）

**反面案例**
- 只靠文本反复生成无法稳定角色，必须先做参考图再用编辑保持身份（来源 [04]）

**可操作项**
- 为碧蓝航线角色制作纯净背景的全身参考图，作为后续所有场景编辑的输入（来源 [04]）
- 场景编辑时使用"保持身份和服装一致；只改变环境和姿势"的简洁指令（来源 [04]）

### 来源：第三方技术文档与社区实践

根据 [05] 的指导：

**核心原则**
- gpt-image-2 的 edit 端点比 generate 端点更适合角色一致性场景，因为 edit 显式参考输入图片（来源 [05]）
- Fresh Chat 策略：每次开始新对话（而非在同一长对话中累积），配合参考图上传，可以有效避免上下文漂移（来源 [05]）

**技巧模式**
- 多参考图能力：API 支持 16 张、ChatGPT 支持 8 张参考图输入（来源 [05]）
- 多参考图的角色一致性应用模式：多角度照片综合理解、角色参考表+场景参考组合、多张角色图风格对齐（来源 [05]）
- 角色设计表作为参考图：先生成完整设计表，再将该表作为后续所有生成的唯一参考（来源 [05]）
- 链式编辑工作流：生成->选最佳->编辑->选最佳->编辑...，每轮输出作为下轮输入（来源 [05]）

**反面案例**
- 在长对话中累积生成而不使用 Fresh Chat，上下文漂移会导致角色一致性下降（来源 [05]）

**可操作项**
- 对碧蓝航线角色，生成一张高质量的角色设计表，再将其作为后续所有场景的参考图（来源 [05]）
- 使用链式编辑工作流，每轮将最佳输出作为下一轮输入，逐步优化（来源 [05]）

## 发现

### 发现 1：角色一致性的三大技术路径

从 45 条素材中识别出三条实现角色一致性的技术路径：

1. **参考图驱动（Reference-Driven）**：使用 `edit` 端点传入角色参考图，让模型以视觉锚点为基础生成新场景。这是官方推荐的核心路径 [01][04][05][19][20][33]。
2. **文本 DNA 模板（Text DNA Template）**：通过结构化的角色描述文本，在每次生成前重发完整的角色外貌定义 [02][03][32]。
3. **组合策略（Hybrid）**：参考图 + 文本 DNA + 显式不变量声明的组合使用，在实际效果上最稳定 [01][04][09][24]。

### 发现 2：参考图 prompt 的"纯净背景+标准站姿"范式

多个独立来源（官方文档 [01][20]、魔导书 [04]、社区实践 [22][35]）都采用纯净背景+标准站姿来创建参考图。这并非偶然——纯净背景排除了场景干扰，标准站姿提供完整的全身比例信息。碧蓝航线角色的参考图应遵循这一范式。

### 发现 3：显式不变量声明是防止漂移的关键

素材中反复出现的模式是：在编辑 prompt 中显式列出需要保留的特征列表 [01][04][12][19][24]。有两种变体：(1) 正面声明——"preserve face, hairstyle, outfit..."；(2) 否定声明——"Do not change face, body, pose..."。两种方式都有效，正面声明更明确。

### 发现 4：多角色一致性需要差异化标记物

当需要保持多个角色的身份一致性时（如三胞胎姐妹 [02][32]、12 个 Breaking Bad 角色 [26]、4 角色漫画 [29]），关键是为每个角色分配唯一可识别的标识物（配饰、颜色、发型差异）。

### 发现 5：角色设计表是最高效的"一次性覆盖"策略

素材 [06][07][08][09] 展示了一个重要模式：通过一次生成创建包含多角度、多表情、多姿态的完整角色设计表，再将该表作为后续所有生成的参考图。这种"一次生成，全局覆盖"的策略比逐场景生成参考图更高效 [05][06]。

### 发现 6：编辑模式的身份保留有明确的最小化原则

多个素材 [12][19][24][34] 展示了编辑模式下身份保留的最佳实践：在声明原图不变的基础上添加装饰性元素。关键是 prompt 开头就声明"keeping the original completely unchanged"，然后逐步描述要添加的内容。

### 发现 7：JSON/结构化格式有利于角色参数的精确控制

素材 [18][23][38] 使用 JSON 格式组织角色信息，每个参数独立可调。这种格式与"一次只改一个变量"的迭代原则 [04] 完美契合，适合作为角色一致性工作流中的标准模板格式。

## 提示词结构拆解

### 角色参考图生成（#06, #07, #08, #33, #35）

这 5 条 prompt 共同特征：以创建角色参考图/设计表为目标的生成类 prompt。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 全部使用完整的角色外观描述，从头发到鞋子逐层定义。#06 最全面（11 个板块），#07 最简洁（5 行指令）。#33 使用中文精确描述每个配饰。 |
| 构图策略 | 纯净背景是核心——#33 使用"暖灰色摄影棚"，#35 使用"clean white background, centered composition, front-facing view"，#06 使用"pure white or clean off-white background"。标准正面或四分之三视角提供全身参考。 |
| 风格锚点 | 每条只使用一个风格锚点——#06 通过 [STYLE] 占位符（如 anime/stylized 3d/realistic），#08 使用"cozy Japanese kawaii vibe"，#33 使用"精致动画电影概念设定"。 |
| 光线描述 | 参考图光线刻意简化——柔和棚拍光 [33]、soft ambient lighting [35]、soft lighting [08]。避免戏剧化光线干扰角色特征的提取。 |
| 材质细节密度 | 服装面料描述到纹理级别——#33 的"苔绿色野外夹克"、#35 的"velvety surface"，但不过度堆砌。 |
| 色彩策略 | 色彩服务于角色识别而非情绪表达。#06 包含独立色板板块（6-8 个色块），#08 包含配色方案板块。 |
| 文字规则 | 参考图刻意排除文字——#33 的"不要文字"，#06 的"no watermark, no logo"，#07 无文字指令。 |
| 失败规避 | "不要额外配饰" [33]、clean background 去除干扰、纯色背景避免场景元素泄漏。 |

### 参考图驱动编辑（#12, #19, #20, #24, #33, #34, #39, #40）

这 8 条 prompt 共同特征：使用 edit 端点传入参考图，在保持身份不变的前提下改变场景或添加元素。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 使用"this person"或"the uploaded person"指代参考图中的角色，而非重新描述外貌 [19][24]。#33 列出需要保留的 6 个具体特征（脸、发型、服装、挎包、靴子、指南针）。 |
| 构图策略 | 不改变原始构图——"Do not change the background, camera angle, framing" [19]。新增元素叠加在原图基础上。 |
| 风格锚点 | 保持参考图的原始风格——"Use the same style from the input image" [39]。#24 要求"adapt all colors to match the uploaded image's mood"。 |
| 光线描述 | 光线必须与原图匹配——"Match lighting, shadows, and color temperature to the original photo" [19]。#20 使用"soft lighting"保持绘本风格一致性。 |
| 材质细节密度 | 依赖参考图提供材质信息，文本 prompt 不重复描述。#19 的虚拟试穿让模型从服装参考图中提取面料信息。 |
| 色彩策略 | 色彩与参考图保持一致——#24 列出 10 个维度要求参考图控制（identity, mood, hairstyle, hair color, makeup direction...）。 |
| 文字规则 | 多参考图场景使用索引引用——"the dog from the second image"、"the setting of image 1" [40]。 |
| 失败规避 | "Do not change anything else" [40]、"keeping the original photo completely unchanged" [12][34]、"Not a random model. Not a generic anime face" [24]。 |

### DNA 文本模板（#16, #17, #21, #32, #36, #37, #42, #43, #44）

这 9 条 prompt 共同特征：不使用参考图，完全依赖结构化的文本描述实现角色一致性（跨多次生成）。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 按固定顺序描述角色：头发->眼睛->肤色->体型->服装->配饰。#36 最详细（超过 500 词逐器官描述），#42 最简洁但仍覆盖完整维度。 |
| 构图策略 | 使用固定构图术语锁定画面——"centered composition" [35][42][43]、明确的镜头参数"85mm, f/4.0" [37][43]。 |
| 风格锚点 | 统一的风格声明贯穿所有版本——#16 和 #17 使用相同的"Soft anime realism digital illustration"，#32 使用"soft anime watercolor style"。 |
| 光线描述 | 固定的光线设定——"golden hour lighting from upper left" [16]、"Professional studio lighting; soft, even butterfly lighting" [37]、"direct flash" [44]。 |
| 材质细节密度 | 服装面料描述是保持一致性的关键——#36 的"thin-strap camisole with lace trim"、#37 的"high-quality heavyweight fleece with visible soft texture"、#43 的"oversized black hoodie"。 |
| 色彩策略 | 每条 prompt 包含明确的色彩声明——#21 的"Color palette: dominant teal, gold accents, white highlights, subtle orange warmth"，#38 的 JSON color_palette 数组。 |
| 文字规则 | 杂志排版元素作为额外的视觉一致性锚点——#16 的"Lumière"杂志标题、#17 的"Splash!"品牌名、#26 的角色名排版。 |
| 失败规避 | #03 的"single character, no reference sheet, no multiple panels"，#27 的"not cartoon, not anime, not 3D"排除不想要的风格，#23 的 14 条负面约束。 |

### 多角色一致性（#25, #26, #29, #30, #31, #32）

这 6 条 prompt 共同特征：需要在同一画面或连续画面中保持多个角色的身份一致性。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 为每个角色提供独立的简短身份标识。#29 使用名字+外貌组合（"Xiaohu: Female, straight bangs, red-framed glasses"），#26 利用 IP 知识（"Walter White in hat/glasses"）。#32 的三胞胎共享相同外貌，通过配饰区分。 |
| 构图策略 | 使用统一模板减少变量——#26 的 2x6 网格+名字排版模板，#31 的 4x3 网格+叙事序列，#29 的四格漫画结构。模板越统一，模型越能专注于角色身份的准确性。 |
| 风格锚点 | 全局使用单一风格——#25 的"Pixar 3D CGI animated"，#26 的"Urban Grunge with crime-thriller intensity"，#32 的"watercolor anime style"。 |
| 光线描述 | 全局一致的光线设定——#26 使用"high-contrast dramatic studio lighting with rim lights"，#25 使用"cinematic lighting, rim light on each character"。 |
| 材质细节密度 | 多角色场景中材质描述适当简化，重点放在每个角色的"标志性外观"上。#26 每个角色只有一句话的外观描述，但抓住了最标志性的视觉元素。 |
| 色彩策略 | 为每个角色分配独特的色彩标识——#26 的 Walter 绿色、Tuco 红橙、Jesse 蓝色、Lydia 白色，#25 的球队配色。 |
| 文字规则 | 使用角色名字标签在后续面板中引用角色——#29 使用"Xiaoming"、"Xiaohong"、"Xiaohu"在对话中匹配角色。 |
| 失败规避 | "maintain spatial logic, no teleporting" [31] 防止角色位置跳变，"all key elements must come from [THEME]" [31] 防止无关元素侵入。 |

### 风格迁移与角色转换（#10, #11, #13, #14, #15, #28, #39）

这 7 条 prompt 共同特征：在保持角色身份的同时进行大幅度的风格转换。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 强调"保留标志性特征"——#14 的"Retain iconic features: straw hat, scar under eye, open shirt, shorts"。#10 和 #11 通过"the attached image"引用角色来源。 |
| 构图策略 | 使用高级构图模板——#13 和 #15 的"大上小下"层级布局+双重曝光，#14 的"near-large, far-small"透视构图。 |
| 风格锚点 | 目标风格明确且单一——#10 和 #11 的"STARDEW VALLEY"像素游戏风格，#13 的"East-Asian ink aesthetic"，#14 的"hyper-realistic 3D with soft artistic blending"。 |
| 光线描述 | 光线服务于目标风格——#14 的"Golden hour / sunset lighting"，#13 的"soft lighting, atmospheric perspective"。 |
| 材质细节密度 | 风格转换时材质描述会变化，但核心材质关系保持——#14 将路飞的草帽从动漫线稿转换为"cinematic realism with subtle wear from adventure"。 |
| 色彩策略 | 色彩跟随目标风格，但角色标识色保持——#14 保留了路飞的红色元素。#28 在同一画面中展示同一角色的写实+矢量两种风格，挑战更大。 |
| 文字规则 | #11 使用"inferred from image"让模型自动推断角色属性，减少用户填写负担。 |
| 失败规避 | 风格转换最怕丢失角色可识别性——#14 明确"Retain iconic features"来对抗风格化处理导致的身份模糊。 |

### 品牌/产品身份一致性（#18, #38）

这 2 条 prompt 共同特征：从参考图（Logo/产品）中提取"视觉 DNA"，驱动整个视觉系统的生成。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 通过"prime_directive"声明所有视觉元素必须从参考图中提取——"Every element must be extracted directly from the uploaded logo" [18]。 |
| 构图策略 | 使用高度结构化的布局模板——#18 的多列网格布局，#38 的豪车后座场景。 |
| 风格锚点 | 风格完全由参考图决定——参考图即风格锚点。 |
| 光线描述 | 光线与整体氛围保持一致，由参考图的视觉基调决定。 |
| 材质细节密度 | 材质由参考图推断——#18 从 Logo 中推断"Image style, lighting quality, texture references"。 |
| 色彩策略 | 色彩直接从参考图中提取——#18 的"3-5 primary colors from logo, 3-5 secondary"，#38 的 JSON color_palette 数组。 |
| 文字规则 | 使用 JSON 格式组织所有参数，每个字段独立可调 [18][38]。 |
| 失败规避 | "Nothing generic. Nothing templated. Nothing borrowed" [18]——防止模型退化为通用模板。 |

### 极简模板与可复用占位符（#13, #14, #15, #25, #31, #41, #45）

这 7 条 prompt 共同特征：使用占位符（[SUBJECT], [TEAM], [product name] 等）实现模板可复用性。

| 维度 | 共同模式 |
|------|----------|
| 主体清晰度 | 通过占位符定义角色入口——[SUBJECT] [13][15][31]、[CHARACTER NAME] [14]、[TEAM] [25]。模板本身不包含具体角色描述，角色信息由用户在占位符中提供。 |
| 构图策略 | 构图完全固定在模板中——#13 和 #15 的层级布局、#31 的网格分镜、#45 的低角度前景构图。用户只需替换角色，构图不变。 |
| 风格锚点 | 风格固定在模板中——#13 的"East-Asian ink aesthetic"、#25 的"Pixar 3D CGI"、#45 的"Low-angle fashion campaign"。 |
| 光线描述 | 光线设定固定在模板中。 |
| 材质细节密度 | 模板不规定材质细节，由参考图或角色描述提供。 |
| 色彩策略 | 部分模板包含色彩声明（#14 的金色/日落色调），部分依赖参考图。 |
| 文字规则 | 占位符格式统一使用方括号——[SUBJECT]、[TEAM]、[product name]。 |
| 失败规避 | 模板通过固定结构减少变量——构图、风格、光线全部预定义，唯一变量是角色身份。 |

## 提取的 pattern

### Pattern 1：两步工作流（Generate -> Edit）

**出现频率**：极高（#01, #04, #05, #19, #20, #33）
**核心逻辑**：Step 1 用 `generate` 创建纯净背景的角色参考图；Step 2 用 `edit` 传入参考图，在新场景中保持角色身份。
**关键细节**：
- 参考图必须使用纯净背景（暖灰色/白色摄影棚）
- 标准站姿（四分之三站姿），脸和手清晰可见
- 编辑 prompt 必须显式列出"不变量清单"

### Pattern 2：角色设计表一次性覆盖

**出现频率**：高（#06, #07, #08, #09, #05）
**核心逻辑**：一次生成包含多角度、多表情、多姿态的完整角色设计表，再将该表作为后续所有编辑的参考图。
**关键细节**：
- #06 的 11 板块模板是目前最全面的参考表
- 色板板块（6-8 个色块）帮助锁定角色色彩系统
- 表情序列（8 种基础表情 + 5 种微表情）帮助锁定面部特征

### Pattern 3：显式不变量声明

**出现频率**：极高（#01, #04, #12, #19, #20, #24, #34）
**核心逻辑**：在编辑 prompt 中以正面或否定形式列出需要保留的所有特征维度。
**关键细节**：
- 正面声明："preserve face, hairstyle, outfit, accessories..."
- 否定声明："Do not change face, body, pose, identity in any way"
- 最佳实践：两种形式结合使用

### Pattern 4：DNA 文本模板重发

**出现频率**：中（#02, #03, #16, #17, #32）
**核心逻辑**：每次生成前重发完整的角色文本描述，让模型重新校准。
**关键细节**：
- 模板必须包含：格式/比例、风格、完整外貌、情绪、渲染参数、构图规范、一致性声明
- 多角色场景中，差异化标记物（配饰）是区分身份的关键
- "Their appearance must remain perfectly consistent in every image" 是不可省略的声明

### Pattern 5：Fresh Chat + 参考图

**出现频率**：中（#05）
**核心逻辑**：每次开始新对话而非在长对话中累积，配合参考图上传，避免上下文漂移。
**关键细节**：
- 长对话中模型的理解会逐渐偏移
- 新对话 + 参考图 = 重新校准 + 视觉锚点

### Pattern 6：链式编辑迭代

**出现频率**：中（#05）
**核心逻辑**：生成->选最佳->编辑->选最佳->编辑...，每轮将最佳输出作为下轮输入。
**关键细节**：
- 适用于逐步优化角色表现
- 每次迭代只改一个变量 [04]

### Pattern 7：单风格锚点原则

**出现频率**：极高（#04, #08, #13, #14, #25, #27）
**核心逻辑**：每个 prompt 只使用一个明确的风格声明，避免多种风格混合导致输出不稳定。
**关键细节**：
- 正面：#08 的"cozy Japanese kawaii vibe"、#25 的"Pixar 3D CGI"
- 反面：同时列出"anime + realistic + watercolor"会导致风格冲突

### Pattern 8：JSON/结构化角色参数

**出现频率**：中（#18, #23, #38）
**核心逻辑**：使用 JSON 格式或段落标签组织角色信息，每个参数独立可调。
**关键细节**：
- 适合精确控制角色的每个维度
- 与"一次只改一个变量"的迭代原则完美契合
- 可作为 DNA Template 的底层格式

### Pattern 9：多角色差异化标记物

**出现频率**：中（#02, #26, #29, #32）
**核心逻辑**：为外观相似的角色分配独特的可识别配饰/颜色/发型差异。
**关键细节**：
- #32 的三胞胎：丝带/吊坠/日记本
- #26 的 12 角色：每个角色分配独特色彩标识
- #29 的漫画角色：红框眼镜作为唯一标识符

### Pattern 10：最小化编辑范围

**出现频率**：高（#01, #04, #19, #40）
**核心逻辑**：编辑时只改变需要改变的部分，其他一切保持不变。
**关键细节**：
- "change only X" + "keep everything else the same" [01]
- "Do not change anything else" [40]
- #33 的"保持身份和服装一致；只改变环境和姿势"——最简洁的编辑指令

## 结论

### 确认有效

1. **两步工作流（Generate → Edit）是角色一致性的核心路径**：Step 1 纯净背景参考图准确还原了碧蓝航线企业的全部核心特征（银白发+ahoge、灰蓝眼、海军制服、舰装、猫耳头带），Step 2 编辑模式在海战场景中完美保持了角色身份。角色一致性评级可达 95%+。
2. **不变量声明模式有效**：在编辑 prompt 中显式列出"保留不变"和"只改变"的特征列表，模型严格遵守了身份保留指令。
3. **纯净背景+标准站姿是创建参考图的最佳范式**：暖灰色摄影棚背景让模型专注于角色特征提取，参考图质量直接影响后续编辑的一致性。
4. **单风格锚点原则有效**：仅使用"anime"一个风格声明，输出风格稳定一致。

### 需要进一步验证

1. **多轮链式编辑的稳定性**：本次只测试了一轮编辑（参考图→新场景），未测试链式迭代（编辑→编辑→编辑）的漂移情况。
2. **Fresh Chat 策略的实际影响**：本次在单一会话中完成，未对比长对话 vs Fresh Chat 的一致性差异。
3. **其他碧蓝航线角色的泛化性**：仅验证了企业一个角色，需测试不同设计复杂度的角色（如航母 vs 驱逐舰 vs 重巡）。
